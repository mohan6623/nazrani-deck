  • No building on the prod box. The box only pulls a ready image, so there is no CPU spike and no downtime blip.
  • Versioned, immutable images. Rollback is re-deploying a previous tag in seconds, not a rebuild.
  • Build once, run the same artifact in UAT and prod. Deterministic, with no "works on the box" surprises.
01 · CI/CD, proposed

Choosing the registry: GHCR vs ECR

GHCRAmazon ECR
Pulls to our EC2small free tier, then billedsame region, free
Auth on the boxGitHub token (new secret)existing IAM role
Ecosystem fitwhere our CI runsnative to our AWS stack
We chose ECR. Both options work, but ECR is the only one where the EC2 pulls images for free (same region) and authenticates with the IAM role it already has. GHCR's free transfer allowance is too small for repeated pulls, after which every pull is billed and crosses the public internet.

01 · CI/CD, proposed

Promotion and hotfixes

Promotion: staging to prod

  • CI builds one image per commit, tagged by git SHA (immutable)
  • That image is deployed to UAT and tested there
  • Production deploys the exact same image that passed UAT, with no rebuild
  • What we tested is bit for bit what ships

Hotfixes

  • Fix on a branch, CI builds a new tagged image, deploy to prod
  • If it misbehaves, roll back to the previous tag in seconds
  • Every image is versioned, so rollback never means a rebuild
The discipline is the same either way: build once, promote the tested artifact, and keep a known-good tag one command away. That is what makes both routine deploys and urgent hotfixes safe.

02 · Cost

Why we are on c7i-flex.large

It was the available choice at launch. When the instance was created, the cheaper burstable options were greyed out in the selection screen. t4g.medium is ARM (Graviton) and would not run our x86 image, and t3.medium was also unavailable in that flow. c7i-flex.large was the practical option left, so that is what we launched on.
  • The result is a compute-optimised instance running a near-idle app, which is why it is the single largest line on the bill.
  • It is not locked in. We can right-size on the existing box with the "Change instance type" action. The options and savings are on the next slide.
02 · Cost

The one optimisation on the table

t3.medium x86, in-place

  • Same 4 GiB RAM, burstable, fits a near-idle app
  • About $30/mo, saves about $27 (47%)
  • In-place change, no rebuild, IP preserved

t4g.medium ARM, later

  • Cheapest. About $16/mo, saves about $41 (72%)
  • Needs an arm64 image rebuild, a small migration
  • Best as a deliberate post-launch move
Why it matters: net is $0 only while credits last. Right-sizing is a five-minute change that halves the biggest line, worth doing before credits expire and before prod. (Open data point: credit balance and expiry, under Billing then Credits.)
DNS

Why we moved DNS to Route 53

The blocker: Amplify serves the site from a CloudFront address, so our root domain nazrani.com has to point at that address. At the apex you cannot use a CNAME, you need an ALIAS record (also called ANAME). GoDaddy's DNS does not offer ALIAS or ANAME records at all, so it simply cannot point the root domain at Amplify. Route 53 supports ALIAS records, so it can. That is the concrete reason we moved.
  • Apex ALIAS records map nazrani.com straight to Amplify and CloudFront, which GoDaddy cannot do.
  • Native integration with Amplify and ACM: automatic DNS validation of our TLS certificates.
  • DNS as code: managed by API and infrastructure-as-code, audited through CloudTrail and IAM.
  • One control plane: DNS lives with the rest of the AWS stack, not in a separate vendor portal.

The domain stays registered at GoDaddy. Only DNS authority moved to a Route 53 hosted zone, so this was low risk with no transfer.
