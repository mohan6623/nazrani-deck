Nazrani Backend - Stage 2

CI/CD pipeline map

Build once in GitHub Actions, store the Docker image in AWS ECR, pull the exact image on EC2, and promote only approved artifacts to production.

Green lane Normal staging to UAT deployment.
Gold lane Hotfix image build and temporary UAT testing.
Blue lane Manual production promotion after approval.
ECR The image registry and rollback source.
01 Normal UAT Flow
Automatic after a deliberate promotion to staging.
Source

Feature branch

Work starts from develop. The branch is tested and gated before integration.

Integrate

Merge to develop

develop stays shippable. Integration is not automatically a release.

Release

Promote to staging

Use git merge --no-ff. No squash on shared branches.

Build

GitHub Actions

deploy.yml builds the Docker image on the GitHub runner.

Store

Push to ECR

Push sha-<short_sha> and update uat-staging.

Run

EC2 pulls image

EC2 logs in with IAM, runs docker-compose.deploy.yml, then checks UAT health.

02 Hotfix Candidate Flow
Builds candidates only. UAT deployment remains manual.
Branch

hotfix/*

Create the hotfix branch from main, not from active development.

Build

build-hotfix.yml

GitHub Actions builds the image but does not replace UAT automatically.

Tags

Candidate image

Push sha-* plus hotfix-name-sha to ECR.

Manual

Deploy to UAT

Run deploy.yml manually with the exact hotfix image tag.

Restore

Return UAT

After testing, deploy uat-staging to restore the normal staging image.

03 Production Promotion Flow
Manual, exact-image promotion. No rebuild during production release.
Approve

UAT passes

Choose the exact image tag that was tested on uat.api.nazrani.com.

Source

Merge to main

main records the production source truth, but does not auto-deploy.

Manual

Promote workflow

Operator enters image_tag and version.

Health

api.nazrani.com

Production health must pass before release tagging happens.

Release

vX.Y.Z tag

Create the version tag only after the production container is healthy.

Release rules

Build once Production receives an approved image, not a freshly rebuilt one.
No shared-branch squash Use --no-ff so ancestry and UAT tags remain meaningful.
UAT moves fast uat-staging is mutable and always points at the current UAT build.
Production is deliberate Manual approval, exact image, health check, then version tag.

Image tags

sha-* Immutable rollback anchor for every build.
uat uat-staging is mutable and deploys the latest staging image.
hotfix hotfix-branch-sha identifies a candidate image.
vX.Y.Z Production release label, created only after prod health passes.

Stage 2 scope: UAT ECR deployment and hotfix candidate builds. Production promotion depends on the later UAT/prod stack split.
